01 The gap

Good tools for how to design.
Nothing for what to design.

Every design methodology assumes you have already chosen what kind of interface to build. None of them help you make that choice.

Interface type selection happens by intuition, by trend, or by whoever speaks loudest. The design methodology literature has no established framework specifically addressing this decision.

04 The framework

A structured conversation,
in four steps.

Designed to make the interface type decision explicit, documented, and impossible to override by opinion alone.

Values come first. Before any technical question is asked.

01 —
Values
What does your organisation actually stand for? Control, accessibility, sustainability, efficiency, user joy. You rank them. That ranking is the primary filter, everything that follows passes through it.
02 —
Context
11 structured questions about your project. Who are the users? How often? What environment? What happens if something fails? These are not engineering questions, they are design questions.
03 —
Patterns
Does this context resemble any of the five documented failures? Always-on camera in public spaces? Replaces something users already solved? Interrupts without being asked? The framework flags the match.
04 —
Report
Not a verdict. A structured report showing which interface types align with your context and values, what the tensions are, and which historical cases are relevant. The designer reads it and decides.
05 How the research was built

Three phases.
One direction.

The framework was not designed from intuition. It was built from literature, tested against history, and validated with real practitioners.

01 —
Framework construction
A review of the design methodology literature found no established framework specifically addressing interface type selection as a standalone decision. Expert interviews with 15 designers and product managers produced a values taxonomy and assessment structure.
02 —
Retrospective validation
Applied the framework to all five documented failures using only information available before each product launched. Target: identify 70% of documented failure factors (Landis & Koch, 1977). Result: 82%.
03 —
Proof-of-concept implementation
Built NEXUS and tested it with 15 designers and product managers. Validated against failures and successes alike. The framework works in practice, not just on paper.

08 How the framework weighs decisions

11 factors.
One clear direction.

Built on DIKW methodology. Every weight is transparent. Each factor comes from real cases where ignoring it cost millions.

28.6%
Values ranking
Primary filter. Assessed before any technical question, prevents building against your own principles.
11.4%
Task complexity
Simple tasks suit voice or automation. Complex ones need screens or AI. Rabbit R1 got this wrong.
9.5%
Frequency of use
High-frequency tasks demand zero friction. Clippy interrupted users who needed full control.
9.5%
Predictability
Predictable tasks can be automated. Unpredictable ones require user control. Humane AI Pin missed this.
9.5%
Context of use
The physical environment constrains what works. Google Glass contradicted the social norms of its context.
9.5%
Information type
Structured data maps to screens. Unstructured queries map to AI. Mismatch creates cognitive overhead.
7.6%
User demographics
Age, proficiency, accessibility needs constrain viable types. AR/VR has high cognitive requirements.
4.8%
Exploration mode
Browsing favors visual. Targeted retrieval favors AI or voice. Mismatch drives abandonment.
4.8%
Error consequence
Medical, financial, legal errors need explicit confirmation. This factor can override all others.
2.9%
Control preference
Manual control vs automation. Captures culture, trust in systems, and regulatory audit needs.
1.9%
Geography
GDPR and EU AI Act restrict AI interfaces in high-risk sectors. Flags compliance risks before a line of code is written.
DIKW methodology
Built on Ackoff (1989)
Values first, data second, information third, wisdom last.
